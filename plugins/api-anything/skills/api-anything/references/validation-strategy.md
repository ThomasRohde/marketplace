# Validation Strategy

This document defines the 7-layer validation framework for CLIs generated by `api-anything`. The framework is inspired by [CLI-Anything](https://github.com/HKUDS/CLI-Anything), adapted for REST/OpenAPI-generated CLIs.

The core philosophy: a generated CLI is not done when it compiles. It's done when it has been proven trustworthy for agent use through multi-layered verification. Agents cannot improvise around broken CLIs — they need deterministic, tested behavior.

## Layer 1: Static validation

Verify the spec and generated code without executing anything.

### Tests to generate

```
test_spec_parses_without_errors
test_all_refs_resolve
test_no_circular_refs_in_schemas
test_no_naming_collisions_across_commands
test_all_operations_have_operation_ids
test_unsupported_constructs_are_surfaced (websockets, callbacks)
test_auth_schemes_are_recognized
test_all_response_schemas_are_valid
test_path_params_match_path_template
test_required_fields_are_marked
```

### What to check

- [ ] Spec parses as valid OpenAPI 3.x or Swagger 2.0
- [ ] All `$ref` references resolve to existing definitions
- [ ] No circular reference chains that would cause infinite recursion
- [ ] Every operation has a unique `operationId` (generate if missing, but flag)
- [ ] No two commands map to the same name after normalization
- [ ] Every path parameter has a matching `{param}` in the path template
- [ ] Required request body fields are marked
- [ ] Unsupported constructs are listed (not silently dropped)

## Layer 2: Contract validation

Verify the generated CLI's interface contract matches the CLI-MANIFEST.

### Tests to generate

```
test_every_command_returns_json_envelope
test_envelope_has_schema_version
test_envelope_has_request_id_as_uuid
test_envelope_has_ok_boolean
test_envelope_has_command_field
test_error_responses_have_error_code
test_error_codes_are_from_registry
test_exit_code_zero_on_success
test_exit_code_nonzero_on_failure
test_exit_codes_match_error_categories
test_guide_command_returns_valid_json
test_guide_lists_all_commands
test_guide_includes_auth_info
test_guide_includes_examples
test_spec_command_returns_api_metadata
test_help_flag_works_on_every_command
test_version_flag_returns_version
test_dry_run_flag_exists_on_write_commands
test_force_flag_exists_on_destructive_commands
test_format_flag_accepts_json_and_table
test_llm_true_forces_json_output
test_piped_output_defaults_to_json
```

### What to check

- [ ] `guide` returns a parseable JSON guide covering all commands
- [ ] Every command's stdout is valid JSON matching the envelope schema
- [ ] `ok: true` commands return exit code 0
- [ ] `ok: false` commands return non-zero exit codes from the defined table
- [ ] Error codes are stable strings from the defined registry
- [ ] `--help` works on every command and subcommand
- [ ] `--version` returns version and schema_version
- [ ] `--format json` produces JSON; `--format table` produces human-readable
- [ ] `LLM=true` environment variable forces JSON output
- [ ] Piped stdout (non-TTY) defaults to JSON
- [ ] Write commands accept `--dry-run`
- [ ] Destructive commands require `--force` in non-interactive mode

## Layer 3: Synthetic behavioral testing

Exercise the CLI logic with synthetic inputs. No real API calls — use fixtures and mocks.

### Tests to generate

```
test_list_command_with_no_filters
test_list_command_with_pagination
test_list_command_with_all_filters
test_get_command_with_valid_id
test_get_command_with_missing_id
test_create_command_with_valid_data
test_create_command_with_missing_required_fields
test_create_command_with_dry_run
test_update_command_with_partial_data
test_delete_command_with_dry_run
test_delete_command_with_force
test_delete_command_without_force_in_non_interactive
test_nullable_fields_handled_correctly
test_enum_fields_validated
test_array_fields_accepted
test_nested_object_fields_accepted
test_file_upload_if_applicable
test_long_running_operation_polling
test_bulk_operation_handling
test_empty_result_set_returns_empty_array
```

### Edge cases to cover

- Pagination: first page, middle page, last page, empty page
- Nullable fields: null vs. absent vs. empty string
- Enums: valid values, invalid values, case sensitivity
- Arrays: empty, single item, multiple items
- Nested objects: flat, one-level deep, deeply nested
- File uploads: small file, large file, missing file
- Long-running operations: immediate completion, polling required, timeout
- Unicode: non-ASCII characters in field values, resource names

### Test fixture specs

Generate at least three fixture specs:

1. **Minimal** — 2-3 endpoints, simple CRUD, API key auth. Tests the basic path.
2. **Medium** — 10-15 endpoints, multiple resources, OAuth2 auth, pagination, enums, nullable fields. Tests realistic complexity.
3. **Messy** — incomplete operationIds, missing auth schemes, inconsistent naming, unresolved refs, deprecated endpoints. Tests robustness with real-world spec quality.

## Layer 4: Integration testing

Run the CLI against a mock HTTP server that simulates the target API.

### Mock server requirements

The mock server must:
- Respond to the exact endpoints defined in the spec
- Return realistic response bodies matching the spec's response schemas
- Simulate auth: reject requests without valid credentials, return 401/403
- Simulate rate limiting: return 429 with Retry-After header
- Simulate server errors: return 500 for specific test paths
- Simulate slow responses: add configurable delay
- Log all received requests for assertion

### Tests to generate

```
test_real_http_request_to_list_endpoint
test_auth_header_is_injected_correctly
test_api_key_in_correct_location
test_oauth_token_is_refreshed_on_401
test_rate_limit_429_triggers_backoff
test_server_error_500_returns_api_error_envelope
test_timeout_returns_timeout_error
test_retry_on_503_with_retry_after
test_no_retry_on_post_without_idempotency
test_pagination_follows_next_link
test_request_body_serialization
test_query_param_encoding
test_path_param_substitution
test_content_type_header_is_set
test_custom_headers_from_profile_are_sent
```

### Mock server implementation

Use a lightweight mock server appropriate for the tech stack:
- **Node.js**: nock, msw, or express with fixtures
- **Python**: responses, httpretty, or aioresponses
- **Go**: httptest.Server
- **Rust**: mockito or wiremock

The mock server should be configured from the OpenAPI spec itself — parse the spec, generate response fixtures, and serve them.

## Layer 5: Subprocess verification

Install or build the CLI and invoke it as an external subprocess — exactly as an agent would.

### Tests to generate

```
test_cli_binary_exists_and_is_executable
test_cli_returns_json_to_stdout
test_cli_writes_progress_to_stderr
test_stdout_and_stderr_are_separate
test_exit_code_0_on_successful_read
test_exit_code_nonzero_on_auth_error
test_exit_code_nonzero_on_invalid_input
test_output_is_valid_json_parseable_by_jq
test_guide_output_is_machine_parseable
test_piped_output_is_json (verify isatty detection)
test_llm_true_env_var_forces_json
test_concurrent_invocations_dont_corrupt_output
test_large_response_is_not_truncated
```

### How to run

```bash
# Build/install the CLI
npm install -g . # or pip install -e . or go install ./...

# Invoke as subprocess and capture output
OUTPUT=$(mycli customer list --env test --format json 2>/dev/null)
EXIT_CODE=$?

# Assert
echo "$OUTPUT" | jq -e '.ok == true'
test $EXIT_CODE -eq 0
```

This layer proves the CLI works as a black box. If subprocess tests pass, an agent can use the CLI.

## Layer 6: Live/sandbox verification

Optional. Only run when the user explicitly enables live testing against a real or sandbox API.

### Prerequisites

- User provides sandbox API credentials
- User confirms the sandbox is safe for automated testing
- User accepts that live tests may create/modify/delete real resources

### Tests to generate (when enabled)

```
test_live_auth_succeeds
test_live_list_returns_data
test_live_get_returns_single_resource
test_live_create_dry_run_does_not_create
test_live_create_then_get_then_delete (cleanup after)
test_live_rate_limit_handling
test_live_pagination_works
```

### Safety rules

- Always test `--dry-run` before actual writes
- Always clean up created resources
- Never run against production without explicit user confirmation
- Log all operations for audit
- Use test-specific resource names (prefixed with `_test_` or similar)

## Layer 7: Documentation verification

Verify that documentation matches the implemented CLI behavior.

### Tests to generate

```
test_guide_examples_match_actual_commands
test_readme_examples_are_valid_commands
test_help_text_matches_implemented_flags
test_error_code_docs_match_actual_codes
test_exit_code_docs_match_actual_codes
test_auth_setup_docs_match_actual_flow
```

### What to check

- [ ] Every example in `guide` output runs without error (or at least parses as a valid command)
- [ ] README examples match the actual command syntax
- [ ] `--help` text for each command lists all implemented flags
- [ ] Documented error codes exist in the error code registry
- [ ] Documented exit codes match the exit code table
- [ ] Auth setup instructions work when followed step by step

## Validation report template

After running validation, generate this report:

```markdown
# Validation Report — mycli

## Summary

| Layer | Status | Tests | Pass | Fail | Skip |
|-------|--------|-------|------|------|------|
| 1. Static | PASS | 10 | 10 | 0 | 0 |
| 2. Contract | PASS | 21 | 21 | 0 | 0 |
| 3. Behavioral | PASS | 18 | 17 | 0 | 1 |
| 4. Integration | PASS | 14 | 14 | 0 | 0 |
| 5. Subprocess | PASS | 12 | 12 | 0 | 0 |
| 6. Live | SKIP | — | — | — | — |
| 7. Documentation | PASS | 6 | 6 | 0 | 0 |

## Proven

- All commands produce valid JSON envelopes
- Auth injection works for API key and OAuth2 client credentials
- Dry-run prevents actual mutations
- Exit codes match error categories
- Guide command is complete and machine-parseable
- Subprocess invocation produces correct stdout/stderr separation

## Assumed (not yet tested)

- OAuth2 authorization code + PKCE flow (requires browser interaction)
- mTLS auth (requires certificate infrastructure)
- Live API rate limiting behavior
- Production-scale pagination (>10,000 records)

## Known issues

- File upload endpoint requires multipart handling (deferred to manual testing)
- Webhook callback endpoints are out of scope
```

## Test fixture library

### Minimal spec (Petstore-class)

```yaml
openapi: 3.0.3
info:
  title: Widget API
  version: 1.0.0
paths:
  /widgets:
    get:
      operationId: listWidgets
      summary: List all widgets
      parameters:
        - name: limit
          in: query
          schema: { type: integer, default: 20 }
      responses:
        '200':
          description: Widget list
          content:
            application/json:
              schema:
                type: array
                items: { $ref: '#/components/schemas/Widget' }
    post:
      operationId: createWidget
      summary: Create a widget
      requestBody:
        required: true
        content:
          application/json:
            schema: { $ref: '#/components/schemas/WidgetCreate' }
      responses:
        '201':
          description: Created
  /widgets/{id}:
    get:
      operationId: getWidget
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string }
      responses:
        '200':
          description: Widget detail
    delete:
      operationId: deleteWidget
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string }
      responses:
        '204':
          description: Deleted
components:
  schemas:
    Widget:
      type: object
      properties:
        id: { type: string }
        name: { type: string }
        status: { type: string, enum: [active, inactive, archived] }
    WidgetCreate:
      type: object
      required: [name]
      properties:
        name: { type: string }
  securitySchemes:
    ApiKey:
      type: apiKey
      in: header
      name: X-API-Key
security:
  - ApiKey: []
```

Use this as the "simple fixture" for golden tests and basic contract validation.
